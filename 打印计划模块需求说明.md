## 1. 背景与目标

价签打印是门店日常高频操作。当前改价、上新、补货均需重新打印价签，但缺乏"批量组织 + 按计划打印"的能力，操作员常逐张打印、易漏打、易与旧价签混淆（货签价不符存在合规风险）。

**目标**：提供"打印计划"管理能力，支持把待打印商品按门店/类型组织成计划，统一预览价签并批量打印；调价场景自动归集到当天计划，确保价签及时、准确更新。

---

## 2. 用户与场景

| 角色 | 职责 |
| --- | --- |
| 店长 / 运营 | 创建打印计划、查看计划状态、安排打印、核对价签准确性 |
| 收银 / 理货员 | 按计划加入/移除商品、设置打印份数、执行打印 |

**典型场景**
1. 每日巡检发现需换签 → 手动新建计划，加入商品，批量打印。
2. 后台改价（服务端事件）→ 商品由服务端**自动**归集进入当天"调价"计划，计划内比对价格，旧已打印价签自动重置为待打印。**用户无操作**。
3. 用户在「改价日志」页开启「自动加入调价打印计划」开关 → 商品信息变更（名称/价格/会员价/产地等）**自动**归集进当天调价计划；开关关闭则完全不自动加入（§4.7）。**用户侧自动触发**。

## 4. 详细功能需求

### 4.1 计划列表

**前置条件**：进入「打印计划」页面。
**交互流程**：
1. 页面加载即拉取计划列表（接口 `POST /mnmart/printPlan/list`，见《接口定义》§9.2）。
2. 筛选栏（左→右）：
   - **门店 Picker**：单选，默认"全部门店"，复用企业&门店下拉 `POST /mnmart/dropdown`（详见《接口定义》§8）。
   - **类型 Segment**：全部 / 手动创建 / 调价自动（对应 `type` `manual`/`price-change`）。
   - **状态 Segment**：全部 / 待打印 / 部分打印 / 已完成（对应 `status` `pending`/`partial`/`done`）。
   - **时间 Segment**：全部 / 今天 / 本周 / 本月（对应 `startTime`/`endTime` 按当前日期计算后传给 `POST /mnmart/printPlan/list`，见《接口定义》§9.2），默认"全部"。
   - **关键字搜索**：按 `name` 模糊匹配。
   - 最右侧 **「产品需求说明」** 入口按钮（非正式功能占位）。
3. 列表列：序号、计划名称、门店（`store`）、类型、计划商品（文本摘要：取前 10 个商品名称，前端渲染为「{text}等{totalCount}个商品」）、商品数、待打印、已打印、状态（标签）、创建时间、操作（查看 / 删除）。
4. 点击「查看」→ 打开计划详情（§4.3）；点击「删除」→ 二次确认后调用 `POST /mnmart/printPlan/delete`（详见《接口定义》§9.5）。**仅「待打印」状态的计划允许删除**；「部分打印」「已完成」计划的操作列不显示删除按钮（或置灰禁用），前端按 `status==='pending'` 控制。

**状态标签配色**：待打印（橙 `#fa8c16`）、部分打印（蓝 `#005cf5`）、已完成（绿 `#52c41a`）。

### 4.2 新建计划

**前置条件**：点击「新建计划」或某商品触发自动建计划。
**交互流程**：
1. 弹窗字段：门店（`shopId`，预选当前筛选门店）、类型（`type` `manual`-手动 / `price-change`-调价）。
2. 校验：`shopId` 必选；计划名称 `name` 按"门店 + 类型 + 日期"自动生成（如 `崧泽大道中心店-调价-0701`），也可由前端传入。
3. 提交 `POST /mnmart/printPlan/add`（详见《接口定义》§9.3），创建后进入空计划详情，提示「点击「添加商品」」。

### 4.3 计划详情（核心）

**信息栏**：门店（`store`）、类型、商品数、待打印、创建时间（`createdAt`）。
**备注条**：说明生产日期取值规则（FIFO，见 §5.1）。

**商品表格列（顺序与固定宽度）**：

| 列 | 宽度 | 对齐 | 说明 |
| --- | --- | --- | --- |
| 复选框 | 40 | 居中 | 用于「打印选中」收集 |
| 序号 | 50 | 居中 | 行号 |
| 商品名称 | 220 | 左 | 超长省略号截断 |
| 编码 | 150 | 左 | 商品编码（`barcode`） |
| 规格 | 120 | 左 | 规格（如 500g/袋，`spec`） |
| 价格 | 118 | 居中 | 当前售价 `¥x.xx/单位`（`price`） |
| 原价 | 112 | 居中 | 普通文本显示（**不划线**） `¥x.xx`（`origPrice`） |
| 生产日期 | 130 | 左 | FIFO 取最早批次（`produceDate`）；多批次标角标（如"3批"，取自 `batchCount` 批次数） |
| 状态 | 100 | 居中 | 待打印 / 已打印 标签 |
| 打印份数 | 110 | 居中 | 未打印可 ±（下限1）；已打印显示份数不可改（`printQty`） |
| 操作 | 100 | 居中 | 未打印：`+份` `-份` `移除`；已打印：无操作 |

> 表头对齐方式与数据列严格一致，避免视觉偏移。表格 `min-width:1260px`，窄屏横向滚动。

**按钮栏**：`+ 添加商品`、`全选未打印`（仅勾未打印项）、`移除未打印`（移除所有未打印项）。
**Footer**：`取消`、`打印选中`（提交 `POST /mnmart/printPlan/print`，详见《接口定义》§9.8，已打印项亦可勾选二次打印）。

**实时同步**：加入/删除/修改份数/打印后，底层计划列表的状态、待打印数、已打印数立即重算刷新（前端按《接口定义》§9.4/`§9.2` 重新拉取或本地重算）。

### 4.4 添加商品

**前置条件**：在计划详情点击「+ 添加商品」。
**窗口规范**：宽度 `760px`，列表区固定高度 `380px`（切换分类不抖动），表格 `width:100%` 铺满。

**筛选**：分类下拉（全部分类 + 动态分类，来源 `POST /mnmart/class/display/list` 或商品 `displayClassId`）、关键字搜索（名称/编码，调用 `POST /mnmart/goods/list` 的 `keyword`）。
**数据来源**：复用 `POST /mnmart/goods/list`（详见《接口定义》§8），返回商品列表（`goodsId`/`goodsName`/`goodsCode`/`goodsSpec`/`goodsUnitName`/`goodsListPrice`/`goodsPrice`/`vipPrice` 等）。

**表格列**：复选框(40) · 商品名称(240, 省略) · 编码(130) · 规格(96) · 价格(110) · 状态(100, 已存在计划则显"已添加"且禁用勾选)。

**交互流程**：
1. 勾选商品 → 点「确定加入」→ 提交 `POST /mnmart/printPlan/addGoods`（详见《接口定义》§9.6），状态重算。
2. 已存在于计划的商品禁用勾选并标记「已添加」，避免重复。
3. 分类切换仅过滤列表，不改变窗口高度。

### 4.5 价签打印

**架构说明**：打印为**客户端行为**——客户端直接基于计划项字段本地渲染价签并完成物理打印输出；服务端**不执行打印**，仅接收客户端上报的打印动作并持久化记录。价签打印与称重打码（`code/printCode` 系列）是**不同功能**，本模块不依赖称重打码接口。

**交互流程**：
1. 详情点「打印选中」→ 客户端弹出价签预览：可选价签模板（标准价签 / 促销价签等），直接使用计划项字段（商品名称、价格、原价、会员价、条码、规格、生产日期等）在客户端渲染价签。
2. 客户端本地执行物理打印（如浏览器 `window.print()` 或客户端打印通道），按每项 `printQty` 循环生成对应张数。
3. 打印完成后，客户端调用 `POST /mnmart/printPlan/print`（详见《接口定义》§9.8）**通知服务端记录**本次打印（打印项、份数、模板、时间、操作人）。服务端仅落盘打印记录，**不执行打印**。
4. **已打印商品**可手动勾选进行二次打印（旧价签作废场景），打印后同样上报记录。

**打印份数**：每项按 `printQty` 循环打印；上报记录成功后该项 `printed=true`、`printedAt=当前时间`。

> 价签打印为纯客户端渲染 + 输出，所需字段均来自计划项（详见《接口定义》§6.5 `PlanItem`），无需服务端打码接口；服务端仅通过 `printPlan/print` 记录打印动作。

### 4.6 调价自动入计划（服务端驱动，用户无操作）

**定位**：本能力由**服务端**完成，前端**无入口、用户无感知**。与 §4.7「改价日志自动加入」（用户侧开关触发）是两类独立能力，勿混淆；现调价归集仅由本能力（服务端自动）与规则5 引擎（§5.5）承担。

**触发方**：后台改价、总部调价下发等商品价格变更事件**完成**时，由服务端自动执行归集。

**交互流程（服务端）**：
1. 服务端接收到商品改价事件（携带 `shopId`/`companyId`/最新 `goodsPrice`/`goodsListPrice`/`vipPrice`）。
2. 调用内部归集接口 `POST /mnmart/printPlan/autoAddFromPriceChange`（详见《接口定义》§9.9，仅服务端调用，前端不可见）。
3. 执行 §5.3 调价归集逻辑（三层决策树）：按「当天调价自动计划 → 当天未完成计划 → 新建调价自动计划」选定目标计划，再对商品做价格比对（同价不重复、异价更新并视情况重置已打印）。
4. 前端仅在「打印计划」列表/详情中**被动看到**结果（计划出现或商品更新）。

**说明**：当前 MVP（最小可行产品）采用前端内置浏览器端 Mock（无真实服务端），故该能力未实际触发；对接后端后由后端实现。详见《接口定义》§14 接口 Mock 说明。

### 4.7 改价日志 · 自动加入调价打印计划（开关）

> ✅ **v1.7 调整**：由 v1.5 的「勾选 + 加入按钮」改为**开关（toggle）**模式。开关打开后，商品信息变更**自动**加入调价打印计划；开关关闭后，商品信息变更**不会**自动加入。与服务端自动归集（§4.6）/规则5 引擎（§5.5）并行不悖。

**定位**：「改价日志」页提供全局开关「自动加入调价打印计划」，控制**商品信息变更是否自动归集到调价打印计划**。属于用户侧自动触发机制，区别于 §4.6 的服务端事件触发。

**开关语义**：
- **开关打开（ON）**：当商品发生以下信息变更时，系统**自动**将商品归集进当天「调价自动」计划（`price-change`）：
  - 商品名称变更；
  - 价格变更：**现价**（`price` / 改后价）、**原价**（`origPrice`）；
  - 其他扩展字段变更：**会员价**（`memberPrice`）、**产地**（`origin`）等（监控字段可扩展，见前端 `PP_AUTO_FIELDS`）。
  - 入计划遵循 §5.3 P1→P2→P3 三层决策树；同条码商品按价格比对，异价更新并重置已打印（规则2/4）。
- **开关关闭（OFF）**：商品信息变更**不会**自动加入任何打印计划（仍可在日志页手动勾选「加入选中项」显式加入）。

**交互流程**：
1. 操作栏提供开关「**自动加入调价打印计划**」（状态持久化于 `localStorage`，默认关闭）+ 状态指示 + 按钮「**模拟新改价**」（演示用，模拟一次商品信息变更）+ 按钮「**加入选中项**」（手动，始终加入调价自动计划）。
2. 开关**打开**后：进入页面时静默同步（确保已变更商品已在计划中）；之后每次商品信息变更（含点击「模拟新改价」）即自动归集，并轻提示「已自动加入调价打印计划」；如开关关闭则仅记录变更、不加入。
3. 开关状态变化即时生效并持久化；打开时立即对当前全部已变更商品执行一次归集。

**前置条件**：进入「改价日志」页（独立页面），可按门店、日期筛选当日改价记录。

**数据来源**：复用 `POST /mnmart/goods/listGoodsPriceRecord`（详见《接口定义》§8），入参 `data.companyId`/`data.shopId`/`data.startTime`/`data.endTime`/`data.keyword` + `requestPage`，返回改价记录（`goodsId`/`goodsName`/`goodsCode`/`goodsSpec`/`goodsUnitName`/`goodsListPrice` 原价/`goodsPrice` 改前价/`goodsAfterPrice` 改后价/`operator`/`recordTime`）。

## 5. 业务规则

### 5.1 生产日期（FIFO）
- 商品存在多批次时，价签生产日期取**最早批次**（最临近过期）。
- 多批次共存时，详情表格生产日期旁标注批次数（如「3批」），hover 提示规则。
- 批次数据来源：**`POST /mnmart/printPlan/addGoods` 响应中随计划项一并返回**（详见《接口定义》§9.6，含 `produceDate`/`batchCount`/`batches`）；FIFO 取最早 `produceDate`，批次数 = `batchCount`（= `batches` 数组长度）。原独立批次查询接口 `POST /mnmart/inventory/batchListByGoodsIdCode`（详见《接口定义》§8）**兼容保留**，但打印计划场景不再单独调用，避免冗余请求（需求2）。

### 5.2 计划状态
```
空计划(无商品) ──► 待打印(pending)
存在未打印商品 ──► 部分打印(partial)
全部商品已打印 ──► 已完成(done)
```
- 计划无商品 → **待打印(pending)**
- 存在未打印商品 → **部分打印(partial)**
- 全部商品已打印 → **已完成(done)**

### 5.3 调价归集逻辑（三层决策树，两端共用）
> 本逻辑为「调价归集」的**共用业务规则**，被两类触发方复用：
> - **A. 服务端自动入计划**（§4.6）：后端改价事件自动调用 `autoAddFromPriceChange`，或销售改价 / 商品编辑等永久调价行为触发（§5.5）。
> - **B. 调价归集引擎（规则5 统一入口）**：前端 `ppAddPriceChangeProducts(products)`（index.html）提供，供销售改价 / 商品编辑等永久调价行为调用（§5.5），亦被 §4.7 开关自动触发复用。
> 触发方不同，但归集、价格比对与状态重置规则一致。字段以现有商品体系为准：`price`(售价) / `origPrice`(原价) / `memberPrice`(会员价)。

**第一层：选定目标计划（按优先级）**
1. **P1（规则4）** 当天已存在「调价自动」计划（`type=price-change`，无论待打印 / 部分打印 / 已完成）→ 直接作为目标计划。
2. **P2（规则2）** 否则，当天存在**未完成**计划（`status≠done`，即待打印 / 部分打印）→ 以该未完成计划为目标。
3. **P3（规则3）** 否则（当天仅有已完成计划，或当天无任何计划）→ **新建**「调价自动」计划（`type=price-change`），作为目标。

> 说明：P1 优先于 P2 / P3，确保「调价自动计划」是调价商品的统一归集入口；仅当当天无调价自动计划时，才退而用未完成计划或新建。

**第二层：商品加入 / 更新（目标计划内）**
  - 计划中无同条码商品 → **新增**（待打印，写入最新价）。
  - 计划中有同条码商品，且价格（售价 / 原价 / 会员价）**完全相同** → 不重复加入、不修改**任何信息（含打印状态）**（对应规则1：价格没变则不更新、不重置）。
  - 计划中有同条码商品，且价格**不同** → 是否更新取决于**目标计划类型与状态**（二者共同前提均为「价格变更且与计划中价格不同」）：
    - 目标计划为「调价自动」（`type=price-change`，**无论是否完成**）→ 更新商品信息（最新价）并重置为**待打印**（旧价签作废，需重打）。
    - 目标计划为「用户新增」（`type=manual`）且**计划未完成**（`status≠done`，**以计划状态为准，非商品状态**）→ 更新商品信息并重置为**待打印**。
    - 目标计划为「用户新增」且**计划已完成**（`status=done`）→ 即便价格不同，也**不更新信息、不重置打印状态**。

> 说明：规则1「已打印数据不进行更新」与规则2 / 4「已打印项更新价格并标记待打印」的张力，已由本规则消解——**价格相同则任何信息都不动（规则1）；价格不同则仅当目标计划为「调价自动」或「用户新增且未完成」时才更新信息并重置待打印（规则2 / 4），用户新增且已完成的计划即便价格不同也保持原状**。该判定以「防止打印过期价格」为总目标，且以计划状态（而非商品打印状态）作为用户新增计划的判定依据。

### 5.4 打印份数
- 每项默认 1 份；未打印项可调增减（下限 1）；已打印项显示份数、不可调。
- 打印时按 `printQty` 循环生成价签。

### 5.5 调价行为定义（规则5）
所有**永久性价格调整**行为，均视为调价行为，相关商品须归集进入打印计划（经 §5.3 三层决策树），以确保价签与最新售价一致：
- 后台改价 / 总部调价下发（服务端事件，§4.6）。
- **销售改价**：前台销售环节发生的价格变更。
- **商品编辑**：商品主数据维护中修改了 `price` / `origPrice` / `memberPrice`。
- 其他导致商品售价永久变化的操作（以产品定义为准）。

> 判定标准：是否**写回商品主数据售价**。仅当价格变更持久化到商品主数据时才计入；临时挂牌、未落库的价格试算不在此列。

### 5.6 打印进度计数（规则6）
- 计划打印进度 = **已打印商品数 / 总商品数**（与 §5.2 状态联动：`pendingCount = totalCount - printedCount`）。
- 商品状态翻转时计数同步变化：
  - 待打印 → 已打印：该计划 `printedCount +1`（进度 +1）。
  - 已打印 → 待打印（如 §5.3 价格变动重置）：`printedCount -1`（进度 -1）。
- 计数由打印 / 重置动作触发即时重算，不依赖异步对账。

## 11. 非功能需求
- **持久化**：生产环境数据持久化由服务端负责（见《接口定义》§9），前端不本地落盘；演示态以《接口定义》§6.9 的本地缓存为临时占位。
- **统一与通用**：接口严格遵循《接口定义》§7 通用规范（POST + 统一包体/响应 + 小驼峰 + JWT），并最大化复用《接口定义》§8 现有接口，避免重复定义字段。
- **层级**：打印预览弹窗层级高于计划详情弹窗，避免被遮挡。
- **响应式**：弹窗窄屏按比例缩放（94vw 上限）。
- **性能**：计划项 < 500 条时列表渲染 < 200ms（纯前端）。
- **可观测**：关键操作（打印、删除）建议埋点统计。

## 13. 名词解释
- **原价（`origPrice`）**：商品吊牌/指导价，价签上以普通文本展示（非划线）。
- **FIFO**：先进先出，此处指按批次生产日期（`produceDate`）取最早批次。
- **二次打印**：已打印商品因调价/破损需重打，手动勾选后重新出签。
- **调价自动入计划**：服务端在商品改价完成时**自动**归集到当天最新调价计划，**用户无操作**（见 §4.6 / 《接口定义》§9.9）。
- **改价日志自动加入**：「改价日志」页开关开启后，商品信息变更**自动**归集进调价计划；关闭则不自动加入（见 §4.7 / 《接口定义》§9.10）。
- **调价自动计划**：由 §4.6 服务端自动归集、类型 `type=price-change` 的计划。
