产品需求文档：商品分类管理

Product Requirements Document — Goods Category Management
📅 版本 v1.2 📌 状态 迭代中 👤 产品：产品通 📆 日期 2026-06-30

1 问题陈述

用户痛点

生鲜零售门店的商品种类繁多（蔬菜、水果、肉禽蛋、水产、粮油调味等），收银员和店长在日常运营中面临以下问题：

  • 商品散乱，查找困难：数十上百个 SKU 在没有分类组织的情况下，收银员需要滚动长列表逐一查找，高峰期严重影响结账速度。
  • 分类体系僵化：随着季节更替和经营策略调整，商品分类的优先级会变化。例如夏季水果销量提升时，水果分类应排在前面；冬季火锅食材增加，肉禽蛋分类需要优先展示。
  • 分类无视觉标识：纯文字分类缺乏颜色区分，在商品列表的左侧侧边栏中不够醒目，店员难以快速定位目标分类。
  • 跨页面割裂：在分类管理页面调整了分类后，切换到商品列表页面需要手动重新筛选，操作繁琐。

不解决的代价

  • 收银效率下降：每笔交易中查找商品平均多耗时 3-5 秒，高峰期累计损失可达十几分钟。
  • 店员抵触情绪：商品组织混乱导致操作负担加重，影响系统使用意愿。
  • 运营灵活性受限：无法快速调整分类顺序来响应季节性或促销策略变化。

2 目标 (Goals)

#目标类型衡量方式
G1 店长可在 30 秒内完成一个分类的增/删/改操作 用户体验 操作耗时可用性测试
G2 分类排序调整后即时生效，无需刷新或手动同步 功能完整性 排序保存后商品列表侧边栏自动更新
G3 商品列表按分类筛选的点击次数 ≤ 2 次 操作效率 从打开商品列表到完成分类筛选的点击路径
G4 每个分类有唯一且稳定的颜色标识，排序不影响颜色 视觉识别 颜色与分类 ID 绑定，不随排序改变

3 非目标 (Non-Goals)

#不做的事项原因
N1 多级分类（如「生鲜 > 蔬菜 > 叶菜」） 用户规模有限，一级分类已满足当前需求；多级分类显著增加 UI 复杂度和数据模型变更成本
N2 分类的批量导入/导出 分类数量较少（通常 10 个以内），逐个维护即可；导入导出需求未经验证
N3 分类的生效/失效状态切换 删除已足够；"禁用"分类引入「有分类但不可用」的中间态，增加理解成本
N4 自定义分类颜色 系统自动分配调色板中的颜色，保持视觉一致性；自定义颜色的 ROI 不足以纳入 v1

4 用户故事 (User Stories)

4.1 企业管理员（可读写）

US1 — 作为企业管理员
我希望能够创建新的商品分类（如「预制菜」「烘焙」），
以便将新增品类纳入系统的分类体系，方便所有门店店员查找。
US2 — 作为企业管理员
我希望能够修改已有分类的名称，
以便纠正录入错误或适配经营策略变更（如「酒水饮料」改为「酒水饮品」）。
US3 — 作为企业管理员
我希望能够删除不再需要的分类，
以便保持分类列表的简洁。删除后该分类下的商品自动归入「未分类」而非丢失。
US4 — 作为企业管理员
我希望通过拖拽来调整分类的展示顺序，
以便将高频使用的分类置顶，让所有门店店员更快找到常用商品。

4.2 门店人员（仅查看）

US5 — 作为门店收银员
我希望看到企业管理员设定好的分类体系（含顺序和颜色），
以便在商品列表页面按分类快速筛选商品，提高收银效率。
US6 — 作为门店店长
在分类管理页面查看当前分类的全貌，以及每个分类下的商品数量，
以便了解分类体系是否符合门店运营需求，向企业管理员反馈调整建议。

4.3 共同诉求

US7 — 作为企业管理员 / 门店人员
在分类管理页面点击某个分类卡片时，直接跳转到商品列表并自动筛选为该分类，
以便快速查看该分类下有哪些商品，减少重复操作。
⚠️ 边界用例 — 作为企业管理员
当我删除一个有 N 件商品的分类时，系统明确提示我这些商品将归入「未分类」，
以避免我误以为商品会被一并删除。
⚠️ 边界用例 — 作为企业管理员
当我在排序模式中取消操作时，分类顺序恢复到排序前的状态，
以便我可以在试排之后放弃变更而无需担心数据出错。
⚠️ 边界用例 — 作为门店人员
当我以门店角色登录时，分类管理页面的编辑、删除、排序按钮均不可见，
以避免我误操作修改企业级的分类体系。

5 需求明细 (Requirements)

5.1 数据模型

字段类型说明
idString唯一标识，新增时用时间戳生成（c + Date.now()），系统保留 _uncategorized
nameString分类名称，最长 20 字，不可重复
hiddenBoolean是否隐藏（仅「未分类」为 true）
sortNumber排序值，越小越靠前；「未分类」固定 9999
colorString颜色 hex 值，新增时从未使用的调色板颜色中自动分配

5.2 功能清单

功能优先级描述验收标准
分类列表展示 P0 卡片式网格布局，显示分类名、颜色圆点、商品数量、编辑/删除按钮 卡片按 sort 升序排列；每张卡片显示正确的颜色和商品数量；空状态显示引导提示
新增分类 P0 弹窗输入分类名称，自动分配颜色和排序值 输入名称不为空且不重名；新分类排在列表末尾；颜色从调色板未使用项中选取
编辑分类 P0 弹窗修改分类名称（颜色不可编辑） 新名称不为空且不与已有分类重名（排除自身）
删除分类 P0 二次确认后删除分类，该分类下商品归入「未分类」 有商品时提示「N 件商品将归入未分类」；确认后分类消失，商品 categoryId 更新为 _uncategorized
拖拽排序 P1 进入排序模式 → 拖拽卡片 → 保存/取消；排序时禁用编辑/删除操作 排序模式入口清晰（「排序」按钮）；拖拽有视觉反馈（高亮边框）；取消恢复快照；保存持久化 sort 值
分类颜色标识 P0 每个分类有固定颜色圆点，颜色与分类 ID 绑定 颜色不随排序变化；新增分类自动取未使用颜色；商品列表侧边栏同步显示颜色
商品列表联动 P0 点击分类卡片跳转商品列表并自动筛选；商品列表左侧侧边栏同步分类顺序和颜色 跳转后侧边栏高亮对应分类；表格只显示该分类商品；页面上方显示分类数量统计
排序模式状态管理 P1 进入排序时快照当前 sort 值；保存应用变更；取消恢复快照 取消后 sort 值完全恢复到进入排序前的状态；排序模式中不出现编辑/删除按钮

5.4 角色权限控制

系统已有用户角色体系，分为企业和门店两级。分类管理模块需接入该权限模型：

操作企业角色门店角色说明
查看分类列表✅ 允许✅ 允许两角色均可查看分类卡片、商品数量
点击卡片跳转✅ 允许✅ 允许两角色均可点击分类卡片跳转到商品列表并筛选
新增分类✅ 允许❌ 禁止门店人员不可见「新增分类」按钮
编辑分类✅ 允许❌ 禁止门店人员不可见卡片上的「编辑」操作
删除分类✅ 允许❌ 禁止门店人员不可见卡片上的「删除」操作
排序分类✅ 允许❌ 禁止门店人员不可见「排序」按钮

实现要点

  • 通过全局 CURRENT_USER_ROLE 变量判断当前角色，值域为 'enterprise' | 'store'。
  • 页面渲染时根据角色决定按钮的可见性，而非在前端隐藏后仍可通过 DevTools 绕过。
  • 后续接入后端 API 时，写操作接口需在服务端再次校验角色权限（前端隐藏仅 UX 层面，不可作为安全边界）。
  • 分类数据为企业级共享：所有门店共享同一套分类体系，由企业管理员统一维护。

5.5 「未分类」特殊规则

  • 不可编辑/删除：「未分类」是系统内置分类，卡片显示「系统」标签，不可点击编辑或删除。
  • 不可排序：「未分类」始终排在列表末尾（sort=9999），不参与拖拽排序。
  • 不在分类管理列表中出现：分类管理页面只显示 hidden=false 的分类，「未分类」卡片以虚线样式单独渲染在网格最右侧。
  • 商品兜底：删除分类时，该分类下的所有商品 categoryId 自动改为 _uncategorized。

7 待确认问题 (Open Questions)

#问题责任人是否阻塞备注
Q1 分类数量上限是否需要限制？目前无上限 产品 否 当前场景分类数通常 ≤ 15，暂无限制必要；如未来达到 30+ 再评估性能
Q2 分类数据是否需要持久化存储（后端 API）？目前仅前端内存保存 工程 否 当前为 Demo/MVP 阶段，数据存内存；接入后端后需同步数据模型
Q3 多门店场景下，分类是全局共享还是按门店独立？ 产品 + 业务 否 已确认：企业级共享。所有门店使用同一套分类体系，由企业管理员统一维护。
决策理由：系统存在跨门店的统计功能（如各分类商品的整体销量对比），若门店各自维护分类，将导致：(1) 同一品类在不同门店分类名不一致，汇总统计无法对齐；(2) 企业管理员无法从全局视角掌握品类结构。企业统一分类同时保证了管理统一（一套标准，一处维护）和数据统一（统计口径一致，跨店可比）。
Q4 是否需要分类的「使用统计」（如某个分类被筛选了多少次）？ 产品 否 可作为 v2 的数据驱动优化手段

8 API 接口定义

8.1 接口概览

#方法路径权限说明
1GET /api/v1/categories 企业 / 门店 查询分类列表
2GET /api/v1/categories/{id} 企业 / 门店 查询单个分类详情
3POST /api/v1/categories 企业 新增分类
4PUT /api/v1/categories/{id} 企业 编辑分类名称
5DELETE /api/v1/categories/{id} 企业 删除分类
6PUT /api/v1/categories/sort 企业 批量更新排序
7GET /api/v1/categories/{id}/products/count 企业 / 门店 查询分类下商品数量（用于删除前确认）

8.2 通用约定

8.2.1 基础路径

所有接口前缀：https://api.{domain}/api/v1

8.2.2 认证方式

所有请求须携带 JWT Token，放在 Authorization: Bearer <token> 请求头中。服务端从 Token 解析当前用户的企业 ID（ent_id）及角色（role："enterprise" | "store"）。

8.2.3 统一响应格式

字段类型说明
codeInteger业务状态码，0 表示成功，非 0 表示失败
messageString提示信息，成功时为 "ok"
dataObject | Array | null响应数据体
request_idString请求追踪 ID，用于排查问题

8.2.4 通用错误码

HTTP 状态码code含义
4001001参数校验失败
4011002未认证（Token 缺失或无效）
4031003无权限（门店角色调用写操作）
4041004资源不存在
4091005资源冲突（如分类名称重复）
5001006服务端内部错误

8.2.5 分页约定

列表类接口支持 page（页码，从 1 开始，默认 1）和 page_size（每页条数，默认 50，最大 200）查询参数。响应中通过 data.list 返回数据，通过 data.total 返回总数。
注：分类数量通常较少（≤ 30），当前阶段分类列表接口不分页，一次返回全部。

8.3.6 批量更新排序

方法 & 路径PUT /api/v1/categories/sort
认证✅ 企业（门店角色返回 403）
说明一次性提交所有分类的排序值。「未分类」不参与排序，其 sort 固定为 9999。请求体中的 items 按期望顺序排列，服务端自动分配连续 sort 值（0, 1, 2, ...）。

请求体 application/json：

{
  "items": [
    { "id": "c2", "sort": 0 },
    { "id": "c1", "sort": 1 },
    { "id": "c5", "sort": 2 },
