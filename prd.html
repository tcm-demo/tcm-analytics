打印计划模块 接口定义（API）

本文档定义打印计划模块的数据结构与接口契约，与《打印计划模块需求说明》配套使用：需求背景、功能交互、业务规则见需求说明文档；本文档聚焦数据模型与接口规范。 字段命名对齐现有 /mnmart 接口体系（小驼峰）。凡现有接口已定义的字段（商品 goods*、门店 shop*、批次 batchNo/productionDate 等）一律沿用，不重新发明。

0. 配套说明

  • 需求侧文档：《打印计划模块需求说明》（背景、用户场景、功能交互、业务规则、名词解释）。
  • 演示态说明：当前 MVP 为前端演示态，采用前端内置浏览器端 Mock（无需后端）；同时提供 GitHub Pages 托管的线上静态 JSON Mock 供其他技术人员调试（详见 §14）。
  • 字段命名映射：前端演示态为前端友好采用简化命名，与后端原生字段一一映射，详见 §6.5、§6.6 注。

6. 数据结构

6.1 实体关系

Shop(门店) 1─* PrintPlan(打印计划) 1─* PlanItem(计划项)
Goods(商品主数据,mart体系) 1─* Batch(批次,复用库存批次)
Template(价签模板,前端渲染概念) 1─* PlanItem(打印时引用)

6.5 打印计划 PrintPlan（新增）

字段命名说明（MVP 演示态）：前端与线上静态 Mock（mock/）为前端友好，采用简化命名；后端原生契约按字段语义一一映射，对接真实后端时由前端适配层转换。映射关系：id ↔ printPlanId、name ↔ planName、store ↔ shopName（含 shopId/companyId 租户隔离，演示态省略）、type（manual/price-change）↔ type（int 0/1）、status（pending/partial/done）↔ status（int 0/1/2）。以下以演示态命名为准。

字段 类型 必填 说明
id string 是 计划唯一 ID（对应后端 printPlanId）
name string 是 计划名称（对应后端 planName）
store string 是 门店名称（快照，对应后端 shopName；后端原生另有 shopId/companyId 租户字段）
type string 是 manual 手动创建 / price-change 调价自动（对应后端 int 0/1）
status string 是 pending 待打印 / partial 部分打印 / done 已完成（对应后端 int 0/1/2）
totalCount int 是 商品总数
pendingCount int 是 待打印数
printedCount int 是 已打印数
createdAt string 是 创建时间 yyyy-MM-dd HH:mm:ss
createdBy string 否 创建人
text string 否 计划商品文本摘要（仅 list 列表接口返回）。取计划内前 10 个商品名称以逗号分隔拼接，如 大白菜, 番茄, 黄瓜；列表页「计划商品」列展示时由前端在其后追加 等{totalCount}个商品，最终形如 大白菜, 番茄, 黄瓜等32个商品（totalCount 见本表）。detail 接口不返回此字段（改用完整 items）。
items PlanItem[] 否 计划内商品项（见 §6.6）。仅 detail 详情接口返回完整 items；list 列表接口为减轻响应体积、避免列表铺开，改用 text 文本摘要（见上），不返回 items。

6.6 计划项 PlanItem（新增）

字段命名说明（MVP 演示态）：id ↔ itemId、name ↔ goodsName、barcode ↔ goodsCode、spec ↔ goodsSpec、unit ↔ goodsUnitName、price ↔ goodsPrice、origPrice ↔ goodsListPrice、memberPrice ↔ vipPrice、produceDate ↔ productionDate。演示态无 source 字段（来源由 type=price-change 的计划隐含）。

字段 类型 必填 说明
id string 是 项 ID（对应后端 itemId）
name string 是 商品名称（快照，加入时固化，对应后端 goodsName）
barcode string 是 商品编码 / 价签条码（对应后端 goodsCode）
price decimal 是 售价（快照，对应后端 goodsPrice）
origPrice decimal 否 原价 / 吊牌价（快照，对应后端 goodsListPrice）
memberPrice decimal 否 会员价（快照，对应后端 vipPrice）
unit string 否 单位名（对应后端 goodsUnitName）
spec string 是 规格（快照，对应后端 goodsSpec）
origin string 否 产地
produceDate string 否 生产日期 yyyy-MM-dd（FIFO 取最早批次，多批次时动态计算，对应后端 productionDate）
batchCount int 否 批次数量（多批次角标；需求2 由 addGoods 响应返回）
batches array 否 批次明细 [{ "produceDate": "yyyy-MM-dd", "qty": int }]（需求2 由 addGoods 响应返回）
printQty int 是 打印份数，默认 1，下限 1
printed boolean 是 是否已打印
printedAt string|null 否 最近打印时间 yyyy-MM-dd HH:mm，无则 null

6.7 价签模板 Template（前端渲染概念）

模板选择为前端价签渲染概念；价签由客户端直接基于计划项字段渲染并输出到打印机（与称重打码 code/printCode 无关），模板决定展示元素（price/origPrice/memberPrice/barcode/name/spec）。服务端不参与打印执行。

字段 类型 说明
templateId string 模板 ID，如 tpl-01 标准价签
templateName string 模板名称
elements string[] 包含元素：price origPrice memberPrice barcode name spec

6.9 演示态 Mock（临时占位，非产品方案）

说明：当前前端 MVP（最小可行产品）仅为演示/调试目的，采用前端内置浏览器端 Mock：所有 printPlan/* 读写由 index.html 中的 PPApi 适配层在浏览器内存中完成（默认模式），无需任何后端，GitHub Pages 静态托管下开箱即用。 正式方案：数据持久化由服务端数据库负责（见 §9 接口契约），前端不落盘、以接口读写为准，确保多端一致与审计可追溯。前端人员联调真实接口时，可使用线上静态 Mock（?api=1 开启，数据来自 mock/ 目录托管的 JSON，详见 §14）。

7. 接口通用规范（统一约定）

本章节为全模块统一约定，所有接口（含复用与新增）均遵守。规范取自现有 /mnmart 接口体系。

  • 基础路径：/mnmart。
  • 协议 / 编码：HTTPS，Content-Type: application/json，字符 UTF-8。
  • 请求方法：统一 POST（列表、查询、详情、写操作均用 POST，与现有体系一致）。
  • 鉴权：POST /mnmart/user/login 登录后返回 jwtToken；后续请求在 Header 携带 Authorization: Bearer {jwtToken}（与现有 /mnmart 接口鉴权一致；具体 header 名以网关为准，统一走 JWT）。
  • 请求包体（统一）： json { "data": { "...业务参数..." }, "requestPage": { "currentPage": 1, "pageSize": 10 } }
  • 列表/分页接口带 requestPage；单条查询/写操作可省略 requestPage。
  • 通用上下文：几乎所有接口需携带 companyId、shopId（租户 + 门店隔离）；shopId 缺省表示跨门店/总部视角。
  • 响应包体（统一）： json { "code": 0, "message": "success", "data": [ "...或对象..." ], "pageInfo": { "pageSize": 10, "currentPage": 1, "totalPages": 1, "totalRecord": 1 } }
  • 列表接口：data 为数组，pageInfo 返回分页信息（pageSize/currentPage/totalPages/totalRecord）。
  • 单条/对象接口：data 为对象，无 pageInfo。
  • 成功判定：以 code === 0 为准；message 为提示文案（现有体系成功时可能为 "success" 或具体成功文案，仅作展示，不应用于逻辑分支）。
  • 字段命名：小驼峰（camelCase），与现有 goods*、shop*、batch* 等保持一致。
  • 时间格式：yyyy-MM-dd HH:mm:ss（字符串）；纯日期 yyyy-MM-dd。
  • 金额：decimal（元，两位小数），JSON 数字（如 3.99）。

下列接口已存在于现有 /mnmart 体系，打印计划模块直接复用，字段以 YApi 导出定义为准，不重新定义。此处仅标注用途与本模块对应关系。

接口 方法 本模块用途 关键入参 data 关键返回 data

9. 新增打印计划接口

仅打印计划领域特有的接口需新增，统一置于 /mnmart/printPlan/*（建议归入「商品管理」分组或独立分组）。所有接口遵守 §7 通用规范（POST + {data, requestPage?} + {code, message, data, pageInfo?}）。字段命名沿用现有体系（小驼峰）。

9.1 通用说明

  • 所有写接口需在 data 中携带 shopId、companyId（与现有体系一致）。
  • 业务校验失败返回 code ≠ 0 并附 message 文案（见 §10）。

9.2 计划列表 POST /mnmart/printPlan/list

请求 data | 参数 | 类型 | 必填 | 说明 | | --- | --- | --- | --- | | shopId | string | 否 | 门店 ID，缺省=全部 | | companyId | string | 否 | 企业 ID | | type | string | 否 | manual 手动 / price-change 调价，缺省=全部（MVP 演示态为字符串枚举；后端原生为 int 0/1，见 §6.5 映射） | | status | string | 否 | pending 待打印 / partial 部分 / done 已完成，缺省=全部（后端原生 int 0/1/2） | | startTime | string | 否 | 创建时间起始 yyyy-MM-dd HH:mm:ss，与 endTime 成对，缺省=全部（对应筛选栏「时间」Segment：今天/本周/本月由前端按当前日期算出后传入） | | endTime | string | 否 | 创建时间截止 yyyy-MM-dd HH:mm:ss | | keyword | string | 否 | 计划名称模糊匹配 |

9.4 计划详情 POST /mnmart/printPlan/detail

请求 data：{ "printPlanId": "pl_20260701_a1b2" } 响应 data：完整 PrintPlan（含 items，字段见 §6.6）。

9.5 删除计划 POST /mnmart/printPlan/delete

请求 data：{ "printPlanId": "pl_20260701_a1b2" } 业务说明：仅「待打印」（status='pending'）的计划允许删除。若计划已存在打印记录（status='partial' 部分打印 / status='done' 已完成），服务端拒绝删除并返回业务错误（见 §10 错误码，code≠0 并附 message 说明，如"仅待打印计划可删除"）；前端在列表操作列已对 status≠'pending' 隐藏/禁用删除按钮作为前置校验。 响应 data：null（需求3：以 code === 0 判定业务成功，data 内不含任何冗余字段，不再返回 { "deleted": true } 之类的盘点参数）。

9.6 添加商品到计划 POST /mnmart/printPlan/addGoods

请求 data

{
  "printPlanId": "pl_20260701_a1b2",
  "shopId": "1810933789769904128",
  "companyId": "1810928032239439872",
  "goodsList": [
    {
      "goodsId": "1824610137569304576",
      "goodsName": "有机小白菜",
      "goodsListPrice": 5.50,
      "goodsPrice": 3.99,
      "vipPrice": 3.59,
      "printQty": 1
    }
  ]
}
  • goodsName/goodsListPrice(原价)/goodsPrice(售价)/vipPrice(会员价) 为新增请求字段（需求2）：服务端据此落库商品快照信息，前端无需额外查询商品主数据。 业务校验：
  • goodsId 不存在 → code ≠ 0（提示"商品无效"）。
  • 商品已在计划：按 §5.3 处理（同价不重复、异价更新），非错误。

响应 data：更新后的完整 PrintPlan（items 中每项包含批次快照：productionDate（FIFO 最早批次）、batchCount（批次数）、batches（批次数组），需求2）——批次数据在此随响应返回，前端不再单独调用批次查询接口。

9.7 移除计划项 POST /mnmart/printPlan/removeGoods

请求 data（二选一）

{ "printPlanId": "pl_20260701_a1b2", "itemIds": ["it_001","it_002"] }

或移除全部未打印：

{ "printPlanId": "pl_20260701_a1b2", "removeAllUnprinted": true }

响应 data：null（需求4：删除类接口精简，以 code === 0 判定成功，data 不含冗余字段）。

9.8 打印记录通知（客户端打印完成后上报）POST /mnmart/printPlan/print

⚠️ 本接口不做打印执行：打印由客户端本地完成。本接口仅用于客户端在打印完成后通知服务端记录本次打印动作并回写计划项状态。

请求 data

{
  "printPlanId": "pl_20260701_a1b2",
  "itemIds": ["it_001","it_002"],
  "templateId": "tpl-01",
  "copies": { "it_001": 2, "it_002": 1 }
}

业务说明：服务端按 itemIds 记录本次打印动作（打印项、份数、模板、时间、操作人），回写各计划项 printed=true、printedAt=now；不调用 code/printCode、不执行任何打印输出。

响应 data：null 或最小对象（需求4：精简响应，以 code === 0 判定成功；如确需扩展可返回 { "recordedItems": [...] }，但前端不依赖 data 内容）。

9.9 调价自动入计划（服务端内部接口）POST /mnmart/printPlan/autoAddFromPriceChange

⚠️ 调用方：仅服务端在改价事件完成时调用，前端不直接暴露、无用户入口（对应 §4.6）。用户侧主动/自动加入请使用 §9.10。

请求 data

{
  "shopId": "1810933789769904128",
  "companyId": "1810928032239439872",
  "goodsList": [
    { "goodsId": "1824610137569304576", "goodsPrice": 3.99, "goodsListPrice": 5.50, "vipPrice": 3.59 }
  ]
}

1. 背景与目标

价签打印是门店日常高频操作。当前改价、上新、补货均需重新打印价签，但缺乏"批量组织 + 按计划打印"的能力，操作员常逐张打印、易漏打、易与旧价签混淆（货签价不符存在合规风险）。

目标：提供"打印计划"管理能力，支持把待打印商品按门店/类型组织成计划，统一预览价签并批量打印；调价场景自动归集到当天计划，确保价签及时、准确更新。

2. 用户与场景

角色 职责
店长 / 运营 创建打印计划、查看计划状态、安排打印、核对价签准确性
收银 / 理货员 按计划加入/移除商品、设置打印份数、执行打印

典型场景 1. 每日巡检发现需换签 → 手动新建计划，加入商品，批量打印。 2. 后台改价（服务端事件）→ 商品由服务端自动归集进入当天"调价"计划，计划内比对价格，旧已打印价签自动重置为待打印。用户无操作。 3. 用户在「改价日志」页开启「自动加入调价打印计划」开关 → 商品信息变更（名称/价格/会员价/产地等）自动归集进当天调价计划；开关关闭则完全不自动加入（§4.7）。用户侧自动触发。

4. 详细功能需求

4.1 计划列表

前置条件：进入「打印计划」页面。 交互流程： 1. 页面加载即拉取计划列表（接口 POST /mnmart/printPlan/list，见《接口定义》§9.2）。 2. 筛选栏（左→右）： - 门店 Picker：单选，默认"全部门店"，复用企业&门店下拉 POST /mnmart/dropdown（详见《接口定义》§8）。 - 类型 Segment：全部 / 手动创建 / 调价自动（对应 type manual/price-change）。 - 状态 Segment：全部 / 待打印 / 部分打印 / 已完成（对应 status pending/partial/done）。 - 时间 Segment：全部 / 今天 / 本周 / 本月（对应 startTime/endTime 按当前日期计算后传给 POST /mnmart/printPlan/list，见《接口定义》§9.2），默认"全部"。 - 关键字搜索：按 name 模糊匹配。 - 最右侧 「产品需求说明」 入口按钮（非正式功能占位）。 3. 列表列：序号、计划名称、门店（store）、类型、计划商品（文本摘要：取前 10 个商品名称，前端渲染为「{text}等{totalCount}个商品」）、商品数、待打印、已打印、状态（标签）、创建时间、操作（查看 / 删除）。 4. 点击「查看」→ 打开计划详情（§4.3）；点击「删除」→ 二次确认后调用 POST /mnmart/printPlan/delete（详见《接口定义》§9.5）。仅「待打印」状态的计划允许删除；「部分打印」「已完成」计划的操作列不显示删除按钮（或置灰禁用），前端按 status==='pending' 控制。

状态标签配色：待打印（橙 #fa8c16）、部分打印（蓝 #005cf5）、已完成（绿 #52c41a）。

4.2 新建计划

前置条件：点击「新建计划」或某商品触发自动建计划。 交互流程： 1. 弹窗字段：门店（shopId，预选当前筛选门店）、类型（type manual-手动 / price-change-调价）。 2. 校验：shopId 必选；计划名称 name 按"门店 + 类型 + 日期"自动生成（如 崧泽大道中心店-调价-0701），也可由前端传入。 3. 提交 POST /mnmart/printPlan/add（详见《接口定义》§9.3），创建后进入空计划详情，提示「点击「添加商品」」。

4.3 计划详情（核心）

信息栏：门店（store）、类型、商品数、待打印、创建时间（createdAt）。 备注条：说明生产日期取值规则（FIFO，见 §5.1）。

商品表格列（顺序与固定宽度）：

列 宽度 对齐 说明
复选框 40 居中 用于「打印选中」收集
序号 50 居中 行号
商品名称 220 左 超长省略号截断
编码 150 左 商品编码（barcode）
规格 120 左 规格（如 500g/袋，spec）
价格 118 居中 当前售价 ¥x.xx/单位（price）
原价 112 居中 普通文本显示（不划线） ¥x.xx（origPrice）
生产日期 130 左 FIFO 取最早批次（produceDate）；多批次标角标（如"3批"，取自 batchCount 批次数）
状态 100 居中 待打印 / 已打印 标签
打印份数 110 居中 未打印可 ±（下限1）；已打印显示份数不可改（printQty）
操作 100 居中 未打印：+份 -份 移除；已打印：无操作

表头对齐方式与数据列严格一致，避免视觉偏移。表格 min-width:1260px，窄屏横向滚动。

按钮栏：+ 添加商品、全选未打印（仅勾未打印项）、移除未打印（移除所有未打印项）。 Footer：取消、打印选中（提交 POST /mnmart/printPlan/print，详见《接口定义》§9.8，已打印项亦可勾选二次打印）。

实时同步：加入/删除/修改份数/打印后，底层计划列表的状态、待打印数、已打印数立即重算刷新（前端按《接口定义》§9.4/§9.2 重新拉取或本地重算）。

4.4 添加商品

前置条件：在计划详情点击「+ 添加商品」。 窗口规范：宽度 760px，列表区固定高度 380px（切换分类不抖动），表格 width:100% 铺满。

筛选：分类下拉（全部分类 + 动态分类，来源 POST /mnmart/class/display/list 或商品 displayClassId）、关键字搜索（名称/编码，调用 POST /mnmart/goods/list 的 keyword）。 数据来源：复用 POST /mnmart/goods/list（详见《接口定义》§8），返回商品列表（goodsId/goodsName/goodsCode/goodsSpec/goodsUnitName/goodsListPrice/goodsPrice/vipPrice 等）。

表格列：复选框(40) · 商品名称(240, 省略) · 编码(130) · 规格(96) · 价格(110) · 状态(100, 已存在计划则显"已添加"且禁用勾选)。

交互流程： 1. 勾选商品 → 点「确定加入」→ 提交 POST /mnmart/printPlan/addGoods（详见《接口定义》§9.6），状态重算。 2. 已存在于计划的商品禁用勾选并标记「已添加」，避免重复。 3. 分类切换仅过滤列表，不改变窗口高度。

4.5 价签打印

架构说明：打印为客户端行为——客户端直接基于计划项字段本地渲染价签并完成物理打印输出；服务端不执行打印，仅接收客户端上报的打印动作并持久化记录。价签打印与称重打码（code/printCode 系列）是不同功能，本模块不依赖称重打码接口。

交互流程： 1. 详情点「打印选中」→ 客户端弹出价签预览：可选价签模板（标准价签 / 促销价签等），直接使用计划项字段（商品名称、价格、原价、会员价、条码、规格、生产日期等）在客户端渲染价签。 2. 客户端本地执行物理打印（如浏览器 window.print() 或客户端打印通道），按每项 printQty 循环生成对应张数。 3. 打印完成后，客户端调用 POST /mnmart/printPlan/print（详见《接口定义》§9.8）通知服务端记录本次打印（打印项、份数、模板、时间、操作人）。服务端仅落盘打印记录，不执行打印。 4. 已打印商品可手动勾选进行二次打印（旧价签作废场景），打印后同样上报记录。

打印份数：每项按 printQty 循环打印；上报记录成功后该项 printed=true、printedAt=当前时间。

价签打印为纯客户端渲染 + 输出，所需字段均来自计划项（详见《接口定义》§6.5 PlanItem），无需服务端打码接口；服务端仅通过 printPlan/print 记录打印动作。

4.6 调价自动入计划（服务端驱动，用户无操作）

定位：本能力由服务端完成，前端无入口、用户无感知。与 §4.7「改价日志自动加入」（用户侧开关触发）是两类独立能力，勿混淆；现调价归集仅由本能力（服务端自动）与规则5 引擎（§5.5）承担。

触发方：后台改价、总部调价下发等商品价格变更事件完成时，由服务端自动执行归集。

交互流程（服务端）： 1. 服务端接收到商品改价事件（携带 shopId/companyId/最新 goodsPrice/goodsListPrice/vipPrice）。 2. 调用内部归集接口 POST /mnmart/printPlan/autoAddFromPriceChange（详见《接口定义》§9.9，仅服务端调用，前端不可见）。 3. 执行 §5.3 调价归集逻辑（三层决策树）：按「当天调价自动计划 → 当天未完成计划 → 新建调价自动计划」选定目标计划，再对商品做价格比对（同价不重复、异价更新并视情况重置已打印）。 4. 前端仅在「打印计划」列表/详情中被动看到结果（计划出现或商品更新）。

说明：当前 MVP（最小可行产品）采用前端内置浏览器端 Mock（无真实服务端），故该能力未实际触发；对接后端后由后端实现。详见《接口定义》§14 接口 Mock 说明。

4.7 改价日志 · 自动加入调价打印计划（开关）

✅ v1.7 调整：由 v1.5 的「勾选 + 加入按钮」改为开关（toggle）模式。开关打开后，商品信息变更自动加入调价打印计划；开关关闭后，商品信息变更不会自动加入。与服务端自动归集（§4.6）/规则5 引擎（§5.5）并行不悖。

定位：「改价日志」页提供全局开关「自动加入调价打印计划」，控制商品信息变更是否自动归集到调价打印计划。属于用户侧自动触发机制，区别于 §4.6 的服务端事件触发。

开关语义： - 开关打开（ON）：当商品发生以下信息变更时，系统自动将商品归集进当天「调价自动」计划（price-change）： - 商品名称变更； - 价格变更：现价（price / 改后价）、原价（origPrice）； - 其他扩展字段变更：会员价（memberPrice）、产地（origin）等（监控字段可扩展，见前端 PP_AUTO_FIELDS）。 - 入计划遵循 §5.3 P1→P2→P3 三层决策树；同条码商品按价格比对，异价更新并重置已打印（规则2/4）。 - 开关关闭（OFF）：商品信息变更不会自动加入任何打印计划（仍可在日志页手动勾选「加入选中项」显式加入）。

交互流程： 1. 操作栏提供开关「自动加入调价打印计划」（状态持久化于 localStorage，默认关闭）+ 状态指示 + 按钮「模拟新改价」（演示用，模拟一次商品信息变更）+ 按钮「加入选中项」（手动，始终加入调价自动计划）。 2. 开关打开后：进入页面时静默同步（确保已变更商品已在计划中）；之后每次商品信息变更（含点击「模拟新改价」）即自动归集，并轻提示「已自动加入调价打印计划」；如开关关闭则仅记录变更、不加入。 3. 开关状态变化即时生效并持久化；打开时立即对当前全部已变更商品执行一次归集。

前置条件：进入「改价日志」页（独立页面），可按门店、日期筛选当日改价记录。

数据来源：复用 POST /mnmart/goods/listGoodsPriceRecord（详见《接口定义》§8），入参 data.companyId/data.shopId/data.startTime/data.endTime/data.keyword + requestPage，返回改价记录（goodsId/goodsName/goodsCode/goodsSpec/goodsUnitName/goodsListPrice 原价/goodsPrice 改前价/goodsAfterPrice 改后价/operator/recordTime）。

5. 业务规则

5.1 生产日期（FIFO）

  • 商品存在多批次时，价签生产日期取最早批次（最临近过期）。
  • 多批次共存时，详情表格生产日期旁标注批次数（如「3批」），hover 提示规则。
  • 批次数据来源：POST /mnmart/printPlan/addGoods 响应中随计划项一并返回（详见《接口定义》§9.6，含 produceDate/batchCount/batches）；FIFO 取最早 produceDate，批次数 = batchCount（= batches 数组长度）。原独立批次查询接口 POST /mnmart/inventory/batchListByGoodsIdCode（详见《接口定义》§8）兼容保留，但打印计划场景不再单独调用，避免冗余请求（需求2）。

5.2 计划状态

空计划(无商品) ──► 待打印(pending)
存在未打印商品 ──► 部分打印(partial)
全部商品已打印 ──► 已完成(done)
  • 计划无商品 → 待打印(pending)
  • 存在未打印商品 → 部分打印(partial)
  • 全部商品已打印 → 已完成(done)

5.3 调价归集逻辑（三层决策树，两端共用）

本逻辑为「调价归集」的共用业务规则，被两类触发方复用： - A. 服务端自动入计划（§4.6）：后端改价事件自动调用 autoAddFromPriceChange，或销售改价 / 商品编辑等永久调价行为触发（§5.5）。 - B. 调价归集引擎（规则5 统一入口）：前端 ppAddPriceChangeProducts(products)（index.html）提供，供销售改价 / 商品编辑等永久调价行为调用（§5.5），亦被 §4.7 开关自动触发复用。 触发方不同，但归集、价格比对与状态重置规则一致。字段以现有商品体系为准：price(售价) / origPrice(原价) / memberPrice(会员价)。

第一层：选定目标计划（按优先级） 1. P1（规则4） 当天已存在「调价自动」计划（type=price-change，无论待打印 / 部分打印 / 已完成）→ 直接作为目标计划。 2. P2（规则2） 否则，当天存在未完成计划（status≠done，即待打印 / 部分打印）→ 以该未完成计划为目标。 3. P3（规则3） 否则（当天仅有已完成计划，或当天无任何计划）→ 新建「调价自动」计划（type=price-change），作为目标。

说明：P1 优先于 P2 / P3，确保「调价自动计划」是调价商品的统一归集入口；仅当当天无调价自动计划时，才退而用未完成计划或新建。

第二层：商品加入 / 更新（目标计划内） - 计划中无同条码商品 → 新增（待打印，写入最新价）。 - 计划中有同条码商品，且价格（售价 / 原价 / 会员价）完全相同 → 不重复加入、不修改任何信息（含打印状态）（对应规则1：价格没变则不更新、不重置）。 - 计划中有同条码商品，且价格不同 → 是否更新取决于目标计划类型与状态（二者共同前提均为「价格变更且与计划中价格不同」）： - 目标计划为「调价自动」（type=price-change，无论是否完成）→ 更新商品信息（最新价）并重置为待打印（旧价签作废，需重打）。 - 目标计划为「用户新增」（type=manual）且计划未完成（status≠done，以计划状态为准，非商品状态）→ 更新商品信息并重置为待打印。 - 目标计划为「用户新增」且计划已完成（status=done）→ 即便价格不同，也不更新信息、不重置打印状态。

说明：规则1「已打印数据不进行更新」与规则2 / 4「已打印项更新价格并标记待打印」的张力，已由本规则消解——价格相同则任何信息都不动（规则1）；价格不同则仅当目标计划为「调价自动」或「用户新增且未完成」时才更新信息并重置待打印（规则2 / 4），用户新增且已完成的计划即便价格不同也保持原状。该判定以「防止打印过期价格」为总目标，且以计划状态（而非商品打印状态）作为用户新增计划的判定依据。

5.4 打印份数

  • 每项默认 1 份；未打印项可调增减（下限 1）；已打印项显示份数、不可调。
  • 打印时按 printQty 循环生成价签。

5.5 调价行为定义（规则5）

所有永久性价格调整行为，均视为调价行为，相关商品须归集进入打印计划（经 §5.3 三层决策树），以确保价签与最新售价一致： - 后台改价 / 总部调价下发（服务端事件，§4.6）。 - 销售改价：前台销售环节发生的价格变更。 - 商品编辑：商品主数据维护中修改了 price / origPrice / memberPrice。 - 其他导致商品售价永久变化的操作（以产品定义为准）。

判定标准：是否写回商品主数据售价。仅当价格变更持久化到商品主数据时才计入；临时挂牌、未落库的价格试算不在此列。

5.6 打印进度计数（规则6）

  • 计划打印进度 = 已打印商品数 / 总商品数（与 §5.2 状态联动：pendingCount = totalCount - printedCount）。
  • 商品状态翻转时计数同步变化：
  • 待打印 → 已打印：该计划 printedCount +1（进度 +1）。
  • 已打印 → 待打印（如 §5.3 价格变动重置）：printedCount -1（进度 -1）。
  • 计数由打印 / 重置动作触发即时重算，不依赖异步对账。

11. 非功能需求

  • 持久化：生产环境数据持久化由服务端负责（见《接口定义》§9），前端不本地落盘；演示态以《接口定义》§6.9 的本地缓存为临时占位。
  • 统一与通用：接口严格遵循《接口定义》§7 通用规范（POST + 统一包体/响应 + 小驼峰 + JWT），并最大化复用《接口定义》§8 现有接口，避免重复定义字段。
  • 层级：打印预览弹窗层级高于计划详情弹窗，避免被遮挡。
  • 响应式：弹窗窄屏按比例缩放（94vw 上限）。
  • 性能：计划项 < 500 条时列表渲染 < 200ms（纯前端）。
  • 可观测：关键操作（打印、删除）建议埋点统计。

12. 待确认 / 后续

  • 会员价是否需在计划详情展示（当前仅价签模板使用）。
  • 批量导入计划（Excel / 后台推送）待评估。
  • printPlan/* 接口分组归属（建议「商品管理」或独立分组），以服务端路由规划为准。
  • 已打印项价格变动时的处理（§5.3）：已明确规则（v1.4 消解）——价格相同则任何信息都不动（含打印状态，规则1）；价格不同则仅当目标计划为「调价自动」或「用户新增且未完成（按计划 status，非商品打印状态）」时才更新信息并重置待打印，用户新增且已完成的计划即便价格不同也保持原状。已在 index.html ppAddPriceChangeProducts 落实（含 planUpdatable 判定，规则5 统一入口）。

13. 名词解释

  • 原价（origPrice）：商品吊牌/指导价，价签上以普通文本展示（非划线）。
  • FIFO：先进先出，此处指按批次生产日期（produceDate）取最早批次。
  • 二次打印：已打印商品因调价/破损需重打，手动勾选后重新出签。
  • 调价自动入计划：服务端在商品改价完成时自动归集到当天最新调价计划，用户无操作（见 §4.6 / 《接口定义》§9.9）。
  • 改价日志自动加入：「改价日志」页开关开启后，商品信息变更自动归集进调价计划；关闭则不自动加入（见 §4.7 / 《接口定义》§9.10）。
  • 调价自动计划：由 §4.6 服务端自动归集、类型 type=price-change 的计划。
